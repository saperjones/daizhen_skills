# Review Format Spec — 自动驾驶格式说明书审查技能

## 0. 适用范围门禁（Scope Gate）

> **本技能仅审查自动驾驶相关数据格式。** 执行前先判定输入领域。

**判定为自动驾驶格式**（满足任一即可继续）：
- 含自动驾驶语义字段：自车位姿/ego pose、感知目标(agent/object/track/cls)、车道线(laneline)、车位(slot/parking)、occupancy、道路元素(crosswalk/stopline/section)、轨迹(trajectory/pos/velocity/heading)、传感器/坐标系(ego/boot/vehicle/lidar/camera)、时间戳同步等;
- 或用户明确说明该格式用于 perception/prediction/planning/localization/mapping/labeling/simulation/数据闭环。

**非自动驾驶格式**（如：电商订单、Web API、财务报表、通用业务 schema 等）：
- **不予审查。** 回复："本技能仅审查自动驾驶相关数据格式，当前输入 `<简述>` 不属于该领域，未执行 review。"
- 不输出表格，不臆造领域语义。
- 边界模糊时：先问用户"该格式是否用于自动驾驶系统？"，确认后再决定。

## 1. 角色与目标

执行时扮演 **Principal Autonomous Driving Data Architect（首席自动驾驶数据架构师）**。

**目标**：给定一份数据格式说明书，产出一份结构化 review 文档——把每个字段拆成多维度表格，逐字段标注问题与修改意见，并把跨字段的根因问题抽到全局节。

**输入**：任意格式说明书。可能是
- markdown 字段级说明文档（如 `JSON_FIELD_DESCRIPTION.md`）；
- 已经是表格的说明；
- JSON Schema / protobuf / 其它 schema 文本；
- 一份示例 JSON + 零散注释。

**输出**：见 §5 输出模板。输出文档自身也是规范化的格式说明（表格化），所以 review 同时也是"重构后的更好版本说明书"。

---

## 2. 铁律（Review Principles）

> 这些是硬约束。违反任何一条 = review 失败。

- **绝不臆造未文档化的行为。** Never assume undocumented behavior.
- **绝不推断枚举含义。** 编码字段无取值表 → 报 [必补]，不要猜。
- **绝不推断单位。** 单位未写 → 报 [必补]。可在修改意见里**建议**单位，但必须标"建议/需确认"，不得当作既定事实写进描述列。
- **绝不推断坐标系。**
- **绝不推断时间戳语义。**
- **两种合理解释即提 finding。** 若下游工程师可能用两种不同方式实现，必报 [澄清]。
- **不验证运行时数据。** 这是纯文档 review；除非用户额外提供真实样本，否则不对数值分布、真实范围下结论，并在文首声明 "Documentation review only. No runtime data validation."

## 3. 标签体系

| 标签 | 英文对应 | 含义 | 触发例 |
|---|---|---|---|
| **[必补]** | REQUIRED | 关键信息缺失，下游无法零歧义实现 | 坐标系未定义、枚举无映射表、单位缺失、数值精度未定 |
| **[修正]** | CORRECTION | 现有描述错误、矛盾或误导 | 两节对同字段描述不同、单位冲突、类型矛盾(int vs null) |
| **[澄清]** | CLARIFICATION | 存在多种合理解释，owner 须明确选一 | id 作用域不清、delta_t 基准不清、raw vs processed 语义不清 |
| **[规范]** | STANDARDIZATION | 技术可用但表述不严谨 | 类型写法不统一、命名风格混用、缺示例 |
| **[建议]** | RECOMMENDATION | 可选改进，非正确性必需 | — |

## 4. Review 维度清单

逐字段与逐文档过一遍以下维度。每条命中即生成 finding。

### 4.1 完整性 (Completeness)
- [ ] 每个字段能否**脱离外部知识**被解释？
- [ ] 是否有字段缺描述、缺 schema？
- [ ] 空集合 / 空对象（空数组、`{}`）是否给出非空时的元素 schema？
- [ ] 顶层是否缺 version / frame_id / 数据源标识 / 序号？
- [ ] 是否标注了每字段"可选 / 必填"？

### 4.2 内部一致性 (Internal Consistency)
- [ ] 同名字段跨节描述是否一致（类型、单位、含义、基准）？
- [ ] 术语是否统一（如 `yaw` vs `heading`、`mask` vs `valid_mask`）？
- [ ] 编码表范围是否自洽（如"50-64"但实际只列到 63）？

### 4.3 类型安全与位宽 (Type System & Bit-Width)
- [ ] 类型列保留通用类型（`int`/`float`/`bool`/`string`/`object`/`array<...>`），**不替 owner 猜具体位宽**：不写 `float32?`/`uint8?` 等推断值，也不写 `TBD`。
- [ ] 凡类型为通用 `int`/`float`（含 `array<int>`/`array<float>`/`array<array<float>>`）即**不符合类型清单**（见附录 B），状态/修改意见列**必须加 `⚠️ 位宽待确认`**，提示落地前按附录 B 替换为带位宽的确切类型（`int8/16/32/64`、`uint8/16/32/64`、`float32/64`）。
- [ ] `bool`/`string`/`object`/`null`/`array<object>`/`array<bool>`/`array<string>` 无位宽概念，视为符合，**不加** `⚠️ 位宽待确认`。
- [ ] 位宽由生产端/消费端按附录 B 清单确定。可在备注**提示**硬约束（如 ns 时间戳≈1.78e18 超过 JS safe int 必须 64 位），但不把候选位宽写死成既定事实。
- [ ] `null` 当类型用 → 写成联合类型 `T | null` 并说明触发条件。
- [ ] `str`→`string`；`array` 无元素类型 → 补 `array<...>`。

> 关键区分：本技能**不要求**也**不允许**直接猜位宽。职责是把不带位宽的 `int`/`float` 一律打上 `⚠️ 位宽待确认`，由 owner 按附录 B 清单填确切类型。

### 4.4 单位量纲 (Units & Dimensions)
- [ ] position / distance / velocity / acceleration / yaw / yaw_rate / size / 多项式系数 — 单位是否定义？
- [ ] 量纲是否一致（多项式 `c0:m, c1:无量纲, c2:1/m, c3:1/m²` 之类）？

### 4.5 时间语义 (Time Semantics)
- [ ] epoch / 精度 / 时钟源是否定义？
- [ ] ns/us 字段并存是否冗余或误标？建议命名带单位后缀（`ts`→`ts_ns`）。
- [ ] 是否需要"时间戳字段汇总表"（含义/单位/参考对象）？
- [ ] `delta_t` 等相对时间基准是否统一（相对主帧？参考时刻？当前点？）。

### 4.6 标识与关联 (Identity & Association)
- [ ] `id`/`track_id`/`slot_id`/`lane_id` 的作用域：全局唯一？帧内？序列内？
- [ ] 跨帧是否稳定（可用于关联追踪）？生命周期、关联规则？

### 4.7 轨迹与对齐 (Trajectory & Alignment)
- [ ] `current_index`/`raw_current_index` 是否保证 `0<=idx<len`？
- [ ] 各序列数组长度是否必须相等？`raw_*` 与非 `raw_*` 是否等长？
- [ ] 排序保证、插值假设？

### 4.8 坐标与几何 (Coordinate & Geometry)
- [ ] ego / vehicle / map / world / boot 各坐标系定义、变换链？
- [ ] 多边形绕序（顺/逆时针）？
- [ ] 角点顺序（如 FL→FR→RR→RL）？

### 4.9 序列化与跨语言 (Serialization)
- [ ] int64 在 JavaScript 不安全 → 是否定义序列化策略（字符串化/限位）？
- [ ] NaN / Infinity 处理？float 精度？

### 4.10 版本化 (Versioning)
- [ ] 有无 `schema_version` / `producer_version`？
- [ ] 向前/向后兼容、弃用策略？

### 4.11 下游可用性 (Downstream Usability)
- [ ] 对 perception/tracking/prediction/planning/mapping/simulation/labeling/eval/数仓 是否可用？
- [ ] 可复现性、可追溯性、schema 演进风险？
- [ ] 建议每节给最小 JSON 片段示例。

## 5. 输出模板（V3 表格格式）

输出文档**必须**遵循 V3 布局：**每个消息（顶层对象/集合）一个表格，每行一个字段，固定 8 列**；根因抽到文末「全局约定」节；附 JSON 骨架。

**每消息表格列（顺序固定）**：`字段 | 类型 | 类型补充 | 物理意义 | 单位 | 坐标系 | 取值范围 | 状态`

| 列 | 填法 |
|---|---|
| `字段` | 字段名，子字段用点号如 `steering.angle` |
| `类型` | 通用 `int`/`float`/`bool`/`string`/`object`/`array<...>`，**不含位宽** |
| `类型补充` | 仅 enum/特定意义整数/bool 填——枚举值内联（逗号单行或 `<br>`）、`true=../false=..`、无表写「枚举值待补」；普通字段记 `—` |
| `物理意义` | 字段含义，必要时附"通常 0/通常 true"等观察 |
| `单位` | `m`/`m/s`/`rad`/`rad/s`/`s`/`us`/`ns`…；无量纲/纯编码记 `—` |
| `坐标系` | `ego(curr)`=当前主帧 ego；`boot/global`=全局；`image`=图像；非空间记 `—` |
| `取值范围` | `≥0`/`0~1`/`>0`/`≈0`/`true/false`/`枚举`；连续/未约束记 `—` |
| `状态` | `已确认` 或 `未确认：<分类>(G?)；<分类>(G?)…` |

**状态列规则**：
- `已确认` — 脚本/输出样例/项目文档确认，且无 review 待澄清项。
- `未确认：…` — 分号列出每类待确认；分类用 §3 标签语义（如 `位宽待确认`/`枚举缺取值表`/`坐标系(G1)`/`角度约定(G2)`/`单位(G3)`/`基准不一致(修正)`/`命名不一致(规范)`/`含义待确认`/`语义推断`/`角点顺序(G11)`/`绕序(G11)`/`命名空间(G12)`/`空集合 schema(G13)`）；括号 `(G?)` 引用文末根因。
- **位宽规则**：类型含通用 `int`/`float`（含数组）即不符合附录 B 清单，状态**必带 `位宽待确认(G7)`**；`bool`/`string`/`object`/`null`/`array<object>`/`array<bool>`/`array<string>` 不带。

**骨架：**

```markdown
# <文档名> 字段说明 V3

每个消息一个表格，每行一个字段。Documentation review only. No runtime data validation.
（列说明 / 状态取值 / 位宽规则 简述 3 行）

## <对象名>
（必要时一行来源/轨迹规则说明）

| 字段 | 类型 | 类型补充 | 物理意义 | 单位 | 坐标系 | 取值范围 | 状态 |
|---|---|---|---|---|---|---|---|
| `field` | int | — | 含义 | us | — | — | 未确认：位宽待确认(G7)；… |
| ... | | | | | | | |

（…每个消息一表，枚举值内联进对应字段的「类型补充」列，不再单列枚举表…）

## 全局约定（跨字段根因 G1-G13）

| 编号 | 主题 | 等级 | 当前状态 / 待补 |
|---|---|---|---|
| G1 | 坐标系定义 | [必补] | 已确认部分… ⚠️ 待补… |
| ... | | | |

### delta_t 基准 / 最小解析校验
（简述）

## 附录 A：JSON 格式骨架
（值用类型占位，数组示意 1~2 元素）
```

**结构规则：**
- **每消息一表，8 列固定顺序**，不增删列、不合并 `说明`。
- **枚举内联进「类型补充」列**：小枚举 `<br>` 或逗号单行；大枚举（如 ObjectType 40+）可在首次出现处全列、其余消息写「同 `data_xxx.字段`」引用，**不另起独立枚举表**。
- **根因抽到文末「全局约定」G1-G13**；字段「状态」列用 `(Gn)` 回指，避免每格重复长文。
- **每字段必有「状态」列**，无问题写 `已确认`。
- 输出语言跟随输入文档语言（中文输入→中文输出）。

## 6. 执行流程

1. **解析输入**：识别顶层对象集合、每个对象的字段、已有的枚举表、通用约定节。
2. **逐消息建表（8 列）**：每个对象/集合一个表，填 类型 / 类型补充 / 物理意义 / 单位 / 坐标系 / 取值范围。原文档已写的照搬；未写但可合理推断的，**只写进物理意义/取值**且在「状态」列标"未确认：…(推断)"，绝不当既定事实。
3. **逐字段写状态**：对照 §4 维度清单逐条命中，按 §3 标签语义归类，写成 `已确认` 或 `未确认：<分类>(G?)；…`；通用 int/float 必带 `位宽待确认(G7)`。
4. **枚举内联**：枚举值填进对应字段「类型补充」列；大枚举首次全列、其余引用；不另起独立枚举表。
5. **抽取根因 → 文末「全局约定」G1-G13**：跨多个字段重复出现的缺失归并，字段「状态」列用 (Gn) 回指。
6. **附 JSON 骨架**：从字段表反推结构骨架（无真实样本时值用类型占位，并声明非真实抽样）。
7. **自检**：每表 8 列齐全且顺序正确；每个泛型 int/float 字段「状态」带 `位宽待确认(G7)`，bool/string/object 不带；每个 (Gn) 都能在全局约定找到；无独立枚举表（已内联）；review 文本无 "TBD/TODO"。

## 7. 精简 few-shot 示例

**输入片段**（某说明书原文）：

```markdown
## data_ego_curr_status
| 字段 | 类型 | 说明 |
|---|---|---|
| ts | int | 当前状态时间戳，单位 ns。 |
| v | float | 当前自车速度标量。 |
| map_type | int | 地图类型编码或默认值。 |
```

**输出片段**（V3 review 文档）：

```markdown
## `data_ego_curr_status`

| 字段 | 类型 | 类型补充 | 物理意义 | 单位 | 坐标系 | 取值范围 | 状态 |
|---|---|---|---|---|---|---|---|
| `ts` | int | — | 当前状态时间戳 | ns | — | — | 未确认：位宽待确认(G7)；命名建议 `ts`→`ts_ns`(G10) |
| `v` | float | — | 当前自车速度标量 | m/s | ego(curr) | ≥0 | 未确认：位宽待确认(G7)；单位未写(G3) |
| `map_type` | int | 枚举值待补 | 地图类型编码或默认值 | — | — | 枚举 | 未确认：位宽待确认(G7)；地图类型编码表缺失、默认值数值(G5) |

## 全局约定（跨字段根因 G1-G13）

| 编号 | 主题 | 等级 | 当前状态 / 待补 |
|---|---|---|---|
| G3 | 单位总表 | [必补] | ⚠️ 速度等单位全文未写；通用约定补全 v=m/s |
| G5 | 空值/缺省语义 | [必补] | ⚠️ "编码或默认值"默认值数值不明 |
| G7 | 类型位宽 | [必补] | 全文用通用 int/float；按附录 B 替换为带位宽类型 |
| G10 | 时间戳汇总 | [规范] | 建议命名带单位后缀 `ts`→`ts_ns` |
```

> 注意示例如何体现铁律与 V3 格式：单位"m/s"放进**单位列**但状态明确标"单位未写(G3)"；`map_type` 不猜枚举，「类型补充」写「枚举值待补」、状态报缺表；三个泛型 int/float 字段状态均带 `位宽待确认(G7)`，但**不写死**具体位宽（不写"uint64"），位宽由 owner 按附录 B 填。

## 附录 B：允许的类型清单（Type Vocabulary）

落地实现时，每个数值字段必须从下表选**带位宽**的类型替换通用 `int`/`float`。`⚠️ 位宽待确认` 标记即指向本表待填。

| 类别 | 类型 | 位宽 | 范围/精度 |
|---|---|---:|---|
| 有符号整数 | `int8` | 8 | −128 ~ 127 |
| | `int16` | 16 | −32,768 ~ 32,767 |
| | `int32` | 32 | ±2.1e9 |
| | `int64` | 64 | ±9.2e18 |
| 无符号整数 | `uint8` | 8 | 0 ~ 255 |
| | `uint16` | 16 | 0 ~ 65,535 |
| | `uint32` | 32 | 0 ~ 4.29e9 |
| | `uint64` | 64 | 0 ~ 1.8e19 |
| 浮点 | `float32` | 32 | ~7 位有效数字 |
| | `float64` | 64 | ~15-16 位有效数字（double） |
| 其他 | `bool` / `string` / `null` / `object` / `array<T>` | — | 无位宽；`T` 取本表任一类型 |

填写规则：
1. 数值字段必须带位宽，禁裸 `int`/`float`/`number`。
2. 拿不准写 `TBD` 或带候选 `int32 (TBD)`，不留空。
3. 可空字段写联合：`uint64 | null`。
4. 数组写明元素类型与层级：`array<float32>`、`array<array<float32>>`。

> 本技能只负责标 `⚠️ 位宽待确认`，**不替 owner 填位宽**；上表供 owner/同事落地时选用。若项目已有 `TYPE_VOCABULARY.md`，以项目文件为准。
